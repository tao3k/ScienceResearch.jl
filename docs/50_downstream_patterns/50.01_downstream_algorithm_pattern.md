# 50.01 Downstream Algorithm Pattern

ScienceResearch.jl is intended to be imported by downstream algorithm packages
when they need reproducible research evidence. It does not host those
algorithms itself.

## Pattern

1. The downstream package owns the algorithm, fixtures, production code, and
   domain-specific correctness oracle.
2. ScienceResearch records the generic contract: dataset shape, workload,
   metrics, validation reports, benchmark samples, and research decision.
3. The downstream notebook references a machine-readable experiment manifest
   with `scienceresearch-artifact: <relative-path>`.
4. The owning package promotes the algorithm only after notebook evidence,
   machine-readable manifest data, and production code remain synchronized.

## Example Domains

This pattern applies to graph search, ranking, ontology validation, OCR
routing, parser strategy, and other algorithm-heavy packages. Each downstream
package should keep its own domain API and use ScienceResearch only for the
research lifecycle and publication substrate.

## Phase-3 Dependency Direction

When a downstream package is ready, add ScienceResearch as a normal package
dependency and keep the integration thin:

- call `validate_dataset` and `validate_algorithm` from downstream tests or
  notebooks;
- call `benchmark_experiment` for candidate timing;
- write `write_experiment_manifest` next to the notebook artifact;
- run `validate_notebook_evidence` before publishing the notebook output.

The downstream package remains the production owner. ScienceResearch remains
the evidence lifecycle owner.
