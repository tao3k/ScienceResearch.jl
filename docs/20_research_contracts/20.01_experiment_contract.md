# 20.01 Experiment Contract

The experiment contract is the first stable layer of ScienceResearch.jl. It is
designed for quick but reviewable validation of algorithm ideas.

## Core Objects

- `DatasetSpec`: identifies the data source and optional scale facts such as
  row count or byte size.
- `WorkloadSpec`: describes the workload shape and resource budget being
  tested.
- `MetricSpec`: declares the metrics that every result must report.
- `ExperimentSpec`: binds a dataset, workload, algorithm idea, metrics, and
  parameters into one reproducible research unit.
- `ExperimentResult`: records measured values, notes, and result artifacts.
- `ValidationCheck` and `ValidationReport`: record deterministic dataset and
  algorithm validation checks before a candidate is treated as evidence.
- `BenchmarkReport`: records warmups, repeated runtime samples, latency
  percentiles, allocation observations, and stability warnings for the same
  experiment contract.
- `ResearchDecision`: classifies whether the evidence is ready for downstream
  production implementation or still needs more validation.

## Research Loop

1. Write the hypothesis as an `ExperimentSpec`.
2. Validate dataset shape and algorithm preconditions with
   `validate_dataset` and `validate_algorithm`.
3. Run the idea with `run_experiment` or benchmark it with
   `benchmark_experiment`, including warmups and regression thresholds when
   timing stability matters.
4. Capture the measured values as an `ExperimentResult`.
5. Compare against a baseline with `compare_baseline`.
6. Classify the evidence with `decide_research_promotion`.
7. Write a Markdown artifact with `write_result_artifact` and a machine-readable
   manifest with `write_experiment_manifest`.
8. Reference the manifest from a notebook or downstream research report.

## AgentAutoResearch Fit

Agent-driven research should produce boring, deterministic evidence before it
claims that an algorithm is promising. This contract lets an agent state:

- which data shape was tested;
- which workload was tested;
- which metrics were required;
- what baseline was used;
- whether the candidate improved the baseline;
- which artifact records the result.
- which notebook evidence references the machine-readable manifest.
- whether benchmark evidence is noisy, regressed, stale, rejected, or ready for
  downstream promotion.

## Benchmark Stability

`benchmark_experiment` is intentionally lightweight and does not replace a
domain-specific performance harness. It gives research notebooks a common
contract for:

- warmup runs before measurement;
- measured sample count;
- min, mean, max, p50, and p95 elapsed milliseconds;
- allocation observations per sample;
- optional regression threshold warnings;
- noisy benchmark warnings when elapsed time spread is too high.

Downstream packages can still run heavier benchmark tools. ScienceResearch
records enough evidence to make early algorithm comparisons reviewable.

## Promotion Statuses

`ResearchDecision.status` uses four statuses:

- `:promote`: thresholds and required baseline deltas pass.
- `:needs_more_evidence`: evidence is incomplete or weak but not rejected.
- `:reject`: the caller requested strict rejection for failed thresholds or
  deltas.
- `:stale_evidence`: notebook or manifest evidence is no longer current enough
  for promotion.

The first contract intentionally avoids domain-specific search, OCR, ontology,
or routing concepts. Those belong in downstream packages that import
ScienceResearch.

## Downstream Example Shape

A downstream package can use ScienceResearch to validate a graph search
strategy, OCR routing idea, ontology proof gate, or any other algorithm without
asking this library to own those domains. For example, a downstream graph
search package would define its own fixtures, candidate generator, and
correctness oracle, then use ScienceResearch to record dataset validation,
benchmark samples, metric thresholds, and the notebook artifact that explains
the result.
